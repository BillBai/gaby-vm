# `bench_business` and First Results, 2026-06-08

[Chinese version](gaby-vm-business-bench-2026-06-08.zh-cn.md)

This note records the first business-logic benchmark for the project. It was
created to answer a question the existing benchmarks could not answer: how slow
is gaby-vm on code that looks like iOS business logic?

The trigger was Step 0 in the archived profile roadmap,
`gaby-vm-cache-hotpath-profile-2026-06-02.md` section 8.2. The older `smoke`
benchmark was too small and ALU-only. The `mixed` benchmark had a 68 percent
NEON mix, which did not represent JSON, protobuf, zlib-style parsing, or similar
app logic. This change built the yardstick first. It did not optimize the
execution path.

## Summary

- `bench_business` adds four diagnostic scalar C microkernels compiled into
  self-contained AArch64 functions. Each kernel models a different business
  shape and reports per-kernel numbers.
- `native_business` runs the same bytes directly on an arm64 host CPU and
  provides the native denominator.
- `bench_business --verify` runs cache mode and decoder mode against the same
  expected `x0` result before performance numbers are trusted.

First results on an M4 Pro, `dev-release`, two runs averaged:

| Kernel | Shape | native ns/insn | cache ns/insn | cache cyc/insn @4.4G | slowdown | cache/decoder |
|--------|-------|----------------:|---------------:|---------------------:|---------:|--------------:|
| **hash** | Integer-heavy dependency chain | 0.31 | 5.98 | ~26 | **~19x** | 13.1x |
| **fsm** | Branch-heavy scanner | 0.066 | 6.20 | ~27 | **~94x** | 14.1x |
| **parse** | Branch-heavy decoder | 0.061 | 6.50 | ~29 | **~108x** | 12.9x |
| **struct** | ILP-friendly load/store | 0.034 | 7.14 | ~31 | **~211x** | 12.7x |

Key conclusions:

1. Cache mode is nearly flat at about 6.5 ns/instruction, regardless of the
   business shape. The dominant cost is a fixed per-instruction tax: dispatch
   plus repeated operand extraction.
2. Slowdown mostly comes from the native denominator. `struct` looks worst
   because native runs it extremely fast. `hash` looks best because the serial
   dependency chain also slows native execution.
3. The no-JIT optimization lever is the flat cache-mode ns/instruction cost.
   Whether that reaches a 50x slowdown depends on the native IPC of the
   workload shape.

## 1. Why This Benchmark Exists

The project goal is to improve interpreted execution from roughly 200x slower
than native toward 50x. That target needs a representative denominator. The
existing benchmarks were not enough:

- `smoke`: 32 straight-line ALU instructions. Timer overhead is visible and the
  workload has only one shape.
- `mixed`: generated by VIXL and 68 percent NEON. Its bottlenecks do not match
  normal app business logic.

The native denominator matters. A previous profile showed that the `smoke`
native path reached an idealized ALU peak around 0.057 ns/instruction. Using
that denominator overstates the slowdown for real code. The new benchmark uses
the same compiled business-logic machine code for both the native baseline and
the simulator paths.

## 2. The Four Microkernels

Each kernel lives under `bench/workloads/business/<name>.c`. The build extracts
a single self-contained AArch64 `.text` function and commits it as a `uint32_t`
array header. Separate kernels make optimization work easier because they show
which workload shape benefits.

| Kernel | Models | Pressure point | Diagnostic use |
|--------|--------|----------------|----------------|
| **parse** | Serialized record decoding, varint/TLV style | Data-dependent branches, byte loads, varint continuation loops, wire-type dispatch | Payload decoding shape; branch-heavy dispatch ceiling. |
| **hash** | Integer digest, FNV-1a plus avalanche mixing | Tight multiply/shift/xor loop, few branches, register-resident state | Best-case scalar interpreter shape. |
| **struct** | Struct-array transform | Offset loads/stores, pointer arithmetic, mixed branches | Load/store leaf path and native best-case shape. |
| **fsm** | Branch-heavy tokenizer-style state machine | Unpredictable state plus character-class dispatch | Worst-case interpreter shape and threaded-dispatch motivation. |

All four kernels follow the same self-contained contract:

- No external calls and no `.rodata` references. Inputs are generated inside the
  kernel, and switches compile to compare/branch chains rather than jump tables.
- Writable memory stays on the stack and fits well under `kMinStackSize`.
- Execution is deterministic for a given seed. `x0` is the oracle output.
- Compile flags:
  `--target=aarch64-linux-gnu -O2 -mgeneral-regs-only -fno-jump-tables
  -fno-builtin -ffreestanding -ffixed-x18`.

`-ffixed-x18` matters because Linux treats `x18` as scratch, while Darwin and
iOS reserve it as the platform register. The native baseline executes these
bytes directly in the host process, and real iOS business code does not use
`x18`.

Generation lives in `bench/workloads/business/gen_business_workloads.sh`:

1. Compile.
2. Use `llvm-objdump -r` to prove the body has zero relocations.
3. Extract `.text` with `llvm-objcopy`.
4. Generate a header with bytes, static word count, and tag.

Dynamic instruction counts and expected `x0` values are measured by runtime
simulation and stored by hand in `oracle_data.h`. The generation script never
rewrites the oracle constants.

## 3. Correctness Gate

`bench_business --verify` single-steps each kernel in cache mode and decoder
mode, then compares final `x0` and dynamic instruction count. The first four
kernels matched bit-for-bit:

```text
parse : dynamic=156172  x0=0xc196940487fb10c0   cache==decoder
hash  : dynamic=32856   x0=0x76dd61c78f2a67e3   cache==decoder
struct: dynamic=47827   x0=0x000005cf4397d892   cache==decoder
fsm   : dynamic=667366  x0=0x72279a557be93af8   cache==decoder
```

This applies the same idea as `ShadowRunner` to the benchmark itself: verify the
cache path against the reference decoder before trusting its speed.

## 4. Measurement Setup

```bash
cmake --preset dev-release -DGABY_VM_BUILD_BENCHMARKS=ON -DGABY_VM_BUILD_NATIVE_BASELINE=ON
cmake --build --preset dev-release --target bench_business native_business

./build/release/bench/native_business --seconds 1.0
./build/release/bench/bench_business --mode cache --seconds 2.0
./build/release/bench/bench_business --mode decoder --seconds 1.0
```

Machine: Apple M4 Pro, macOS 26.5, no pinning, P-core roughly 4.4 GHz,
`dev-release`. Each side ran twice and the note uses the average. Run-to-run
variance was in the tens of percent, so treat the first digits as meaningful and
not the tail.

The primary metric is `iterations_per_second`. `ns_per_instruction` is derived
from the committed dynamic instruction count.

## 5. Interpretation

### 5.1 Cache Mode Is Flat, Native Is Not

The cache-mode column stays within 5.98 to 7.14 ns/instruction. The native column
spans almost an order of magnitude, from 0.034 to 0.31 ns/instruction. Since
slowdown is cache divided by native, almost all slowdown variance comes from
native.

- `struct` reaches about 6.8 IPC natively, so the native denominator is tiny and
  the slowdown looks large. The interpreter is not unusually slow there; native
  is unusually fast.
- `hash` has a serial multiply/xor dependency chain. Native cannot exploit much
  ILP, so gaby-vm is already around 19x on that shape.
- `parse` and `fsm` sit around 100x because native branch prediction holds up
  and gaby-vm pays the fixed per-instruction cost.

### 5.2 What the 50x Target Means

If cache mode dropped from about 6.5 ns/instruction to about 2.86 ns/instruction,
as estimated in the previous roadmap for smoke, the four kernels would move to:

| Kernel | Today | With cache at 2.86 ns/insn | Reaches 50x? |
|--------|------:|----------------------------:|--------------|
| hash | 19x | ~8x | yes |
| fsm | 94x | ~42x | yes |
| parse | 108x | ~47x | barely |
| struct | 211x | ~93x | no |

For parse, fsm, and hash, the 50x target is reachable without JIT if the project
reduces the fixed per-instruction tax. `struct` is the hard case because native
executes it near the ILP ceiling of the core.

### 5.3 Optimization Implications

- The benchmark supports the dispatch/operand-predecode roadmap. Since cache
  mode is flat, reducing the fixed per-instruction tax helps every business
  shape.
- `fsm` is the best validation target for threaded dispatch because it magnifies
  branch and dispatch costs.
- The next useful spike is to implement the first dispatch-shrink step, measure
  these four kernels again, then decide whether to proceed to larger threaded
  dispatch or operand-predecode work.

## 6. Limits

- Single machine, no pinning, variance in the tens of percent.
- The kernels are synthetic shapes, not captured app traces.
- This note measures against native only. A Lua/JSC LLInt head-to-head is still
  needed to answer whether gaby-vm beats embeddable interpreters on the same
  semantics.
- `struct` represents an ILP upper bound, not typical business code.
