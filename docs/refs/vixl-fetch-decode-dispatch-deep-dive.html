一条 ADD 触发 ~3 次 member-fn-ptr 调用（树游走）， ~2~3 次虚函数调用（visitor 链）， 1 次 heap 分配（Metadata）， 2 次同名字符串 hash（decoder + simulator）， ~8 次位段重抽（leaf 内部）。 实际改了状态的只有一次寄存器读 + 一次加法 + 一次寄存器写。

3. 性能瓶颈热力

每行一个执行阶段，给每条 ADD 的近似 cycle 估计、是否可摊销、缓存后剩多少。 精度只到 order-of-magnitude，硬掰反而误导。

Leaf 实际语义（add + 寄存器写） ~5–10 不可摊销 ~5–10 —
合计 ~300–500 — ~40–120 —
结论：V1 缓存（仅 dispatch 层摊销，不动 leaf）就能把 dispatch overhead 从 300–500 cycle 压到 40–120 cycle， leaf 真实工作占比从 ~3% 提升到 ~10–25%。 V2 加 operand pre-extraction 可以再砍 30–80 cycle。

3.1 二级效应

  • BTB 压力：每条指令 5~7 次间接调用，多种类指令的 BTB 容量 压力大；缓存后压成 1 次，BTB 压力降一个量级。
  • I-cache 压力：decoder + decoder-constants + visitor + leaf helper 加起来几十 KB 热路径；缓存后只剩 leaf，I-cache 命中率显著提升。
  • D-cache 压力：decode_table_ / form_to_visitor_fn_map_ / visitor list 节点都不再热；D-cache 留给 guest data。
  • Allocator 抖动：每条指令一次 unordered_map heap alloc 消失；长跑测试的 RSS 增长曲线会平整。

4. 缓存设计

4.1 关键前提

pc_ 是宿主指针（simulator-aarch64.h:5348）， GetNextInstruction() 是 this + 4 指针算术 （instructions-aarch64.h:734）—— 所以缓存可以直接以 const Instruction* 为 key，无需任何翻译。

Lookup 是 两次 load + 一次乘移，跟 PC 算术天然 1:1 对齐， cache friendly，回收时直接 free 整个 page bucket。

4.5 热路径伪代码

void Simulator::ExecuteInstructionCached() {
  VIXL_ASSERT(IsWordAligned(pc_));
  pc_modified_ = false;
  MaybeBTypeCheck();  // runtime state machine, 不能 cache

  bool last_instr_was_movprfx = ...;

  DecodedSlot* slot = predecode_cache_.LookupOrInsertEmpty(pc_);
  if (VIXL_LIKELY(slot->state == DecodedSlot::kValid)) {
    form_hash_ = slot->form_hash;          // R1: load-bearing
    slot->leaf_fn(this, pc_);
  } else {
    SlowPathDecodeAndPopulate(pc_, slot);  // 走完整 decoder，回填 slot
  }

  if (last_instr_was_movprfx) { /* MOVPRFX 校验 */ }

  last_instr_ = ReadPc();
  IncrementPc();
  // LogAllWrittenRegisters / UpdateBType / auditor check
}

4.6 V2 拓展（不在 V1 范围）

V1 命中后还要走一次 leaf 内部的 8 次位段重抽。V2 可以在 slot 里加 per-form 联合体，把 Rd/Rn/Imm 等预先抽好，每个 leaf 写一个 fast-path 版本。 成本：每个 leaf 双份代码；好处：再省 30–80 cycle。

5. 风险卡片墙（R1–R12）

按踩中后修复成本从高到低排序。点击展开看触发条件、缓解策略、源码引用。

R1 · form_hash_ 必须在调 leaf 前还原

问题：共享 Simulate_* 入口的 leaf 会读 form_hash_ 选行为；缓存路径调 leaf 前忘了写就会跑错分支。

缓解：form_hash 进 slot， form_hash_ = slot->form_hash 必须在 leaf_fn(...) 之前。 覆盖共享 leaf 的多个 form 写测试。

引用：simulator-aarch64.cc:2306-2323 + simulator-aarch64.cc:2325 (Simulate_PdT_PgZ_ZnT_ZmT)

R2 · MOVPRFX 链式校验

问题：SVE MOVPRFX 要求紧跟一条「能合并的指令」， ExecuteInstruction 末尾会校验上一条 form_hash。

缓解：跨缓存命中也要正确传递 last_instr_ 和 form_hash_；不要省掉 last_instr_was_movprfx 提前计算。

引用：simulator-aarch64.h:1420-1421 + simulator-aarch64.h:1431-1434

R3 · CPU Features Auditor 必须满足

问题：ExecuteInstruction 末尾 VIXL_CHECK(cpu_features_auditor_.InstructionIsAvailable()) 会 abort。绕过 visitor 链就绕过了 auditor。

缓解（任选）：

① Region 级预验证：populate 时跑一次 auditor，结果存 slot； ② 保留 auditor 调用，用预构造 Metadata 不每次重 alloc； ③ 关掉 auditor（破坏安全契约，不推荐）。

引用：simulator-aarch64.h:1441 + cpu-features-auditor-aarch64.h:107-118

R4 · trace / debugger / 自定义 visitor

问题：开 LOG_DISASM 必须先调 PrintDisassembler， debugger 启用要查 breakpoint，第三方 AppendVisitor 注册的也要被调。 缓存路径默认绕过 visitor 链。

缓解：SetTraceParameters / SetDebuggerEnabled / AppendVisitor 时设全局 fallback flag；命中路径开头判 flag 为 true 就走原 decoder。

引用：simulator-aarch64.h:1423-1428 + decoder-aarch64.cc:90-135

R5 · BType / Guarded Page

问题：BType 是 runtime state（每条指令推进）， PcIsInGuardedPage 是配置查询。不能 cache 到 slot。

缓解：循环开头继续执行 MaybeBTypeCheck()。开销可忽略。

引用：simulator-aarch64.h:1408-1418 + simulator-aarch64.h:1396 (PcIsInGuardedPage)

R6 · Self-Modifying Code

问题：VIXL 直接读宿主内存做 fetch；guest 改写 code page 后缓存项就 stale。

V1 缓解：明确假设「guest code immutable」，提供 InvalidateCacheRegion(begin, end)。

V2 缓解：Memory::Write 命中 code region 时自动失效 对应 slot（要求 code region 边界已知）。

引用：simulator-aarch64.h:393-411 (Memory::Read)

R7 · 多实例 / 共享内存

问题：Gaby-VM 计划支持多 Simulator 实例； 全局 cache 要锁 / lock-free，per-instance cache 重复 populate。

V1 推荐：per-instance cache。populate 在 cold path 影响小。

V2：跨实例共享只读的 (form_hash, leaf_fn) 部分， 仍要解决无锁查表。

引用：架构 doc「Threading and determinism」段

R8 · Cache 大小失控

问题：1MB code → 4MB cache，10MB → 40MB，100MB → 400MB。

缓解：per-page bucket 自带 region 粒度回收； mprotect / unmap 时清桶；上限 + LRU 替换；暴露 FlushPredecodeCache()。

R9 · iOS / no-JIT 合规

问题：iOS 不允许 RWX 页 / runtime codegen。 预解码缓存看似「生成了什么」要明确不是 codegen。

结论：缓存项是 ordinary data， leaf_fn 是 .text 段已存在的函数指针。 不分配 executable memory，不调 mprotect(PROT_EXEC)。 完全合规。

R10 · 正确性回归

问题：缓存路径必须 byte-identical 复现非缓存路径的所有副作用 （regs、PSTATE/NZCV、内存、PC、btype_、ffr_register_）。

缓解：开发期开 shadow mode——同时跑 cache 路径和 reference decoder，每条指令后 diff 全部状态，差异即 abort。 性能差 100×，但跑 testsuite 正合适。

R11 · BR / RET / 间接跳转

问题：缓存只能加速「已经 fetch 过的 PC」。 BR x0 / RET / BLR x0 第一次目标必走慢路径。

影响：典型 hot loop 直接跳转居多，命中率高； 函数调用边界的 BLR / RET 二次访问命中。

结论：V1 不解决。V2 考虑 indirect-target predictor 或 basic-block linking。

R12 · unallocated / unimplemented 编码

问题：Decoder 通过 form_to_unalloc_ 检查 + VisitUnimplemented 处理。如果缓存把它们当普通 form 缓存 会跑错。

缓解：slot.state 引入 kUnallocated / kUnimplemented，命中后强制走对应路径，不调 leaf_fn。

引用：simulator-aarch64.cc:3915 (VisitUnallocated) + simulator-aarch64.cc:3907 (VisitUnimplemented) + simulator-aarch64.cc:3896 (VisitReserved)
