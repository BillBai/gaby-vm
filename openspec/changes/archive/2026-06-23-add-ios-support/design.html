在 iOS Simulator 与真机上跑全套 correctness 与 benchmark 的技术方案：Xcode runner、CMake 单一来源、CI 门禁与显式 skip 语义。

本文要点

  1. CI 目前只在 macOS arm64 上跑；库已能交叉编译 iOS arm64，缺的是在 iOS 上执行测试/benchmark 的路径。
  2. iOS 没有「跑 console binary」的等价物；runner 始终是 development build，debugger 附着时 iOS 才允许 MAP_JIT。
  3. 采用提交的薄 Xcode 项目（App + XCTest），库通过 CMake 生成的 Xcode subproject 消费；fallback 是直接编译源文件 + drift check。
  4. 测试/benchmark 抽 callable entry，host CTest 与 iOS XCTest 共用一套逻辑，不 fork 源码列表或 suite 实现。
  5. CI 沿用 portable ci/ 脚本：ios-sim-test 门禁 PR，iOS bench report-only；非 arm64 runner 必须显式 SKIP。
  6. Native baseline 在 iOS 上尽力而为：可用则报告，executable memory 被拒则明确说明未运行。
  7. 迁移是增量式的：先抽 entry → 本地 Simulator 验证 → 接 CI → 稳定后再设为 required gate。

## Context

See [`proposal.md`](proposal.md) for motivation. Today the only execution
target exercised by CI and the test/bench suites is macOS arm64. The library
already cross-compiles for iOS arm64; what is missing is a way to *run* the
suites on iOS and a CI path that does so. CI structure and the deterministic-
gate philosophy are documented in
[`docs/refs/ci.md`](../../../docs/refs/ci.md); the test harness in
[`docs/testing.md`](../../../docs/testing.md).

Two facts shape every decision below:

- The existing tests and benches are standalone `main()` executables driven by
  CTest (host-only). iOS has no equivalent of "run this console binary".
- The iOS runner is always a **development** build, launched from Xcode with a
  debugger attached — the configuration in which iOS permits `MAP_JIT`.

Goals / Non-Goals

在 iOS 上跑全套 suite；单一来源；fit 现有 CI；不改语义、不做 embedder 框架。

Goals

  • 在 iOS 上跑全套 correctness suite 与 benchmark harness——CI 用 Simulator，本地用真机。
  • 库 source list 与 suite/bench 逻辑保持单一来源；iOS 路径不得 fork 任一侧。
  • fit 现有 CI 模型：确定性 correctness 门禁、report-only 性能、逻辑全在 portable ci/ 脚本里。

Non-Goals

  • CI 里自动化真机运行（仅本地）。
  • 修改解释器语义、公开 API 或 VIXL import boundary。
  • 为 embedder 提供可复用的 iOS app 框架——runner 是 test/bench host，不是 shipping artifact。tater 以 source（submodule）方式集成库。

## Goals / Non-Goals

**Goals:**

- Run the full correctness suites and the benchmark harness on iOS — the
  Simulator under CI, a physical device locally.
- Keep one source of truth for both the library source list and the suite/bench
  logic; the iOS path must not fork either.
- Fit the existing CI model: deterministic correctness gate, report-only perf,
  all logic in portable `ci/` scripts.

**Non-Goals:**

- Automated device runs in CI (local-only).
- Changing interpreter semantics, the public API, or the VIXL import boundary.
- A reusable iOS app framework for embedders — the runner is a test/bench host,
  not a shipping artifact. tater integrates the library as source (submodule).

Decisions

五个核心决策：Xcode runner、CMake subproject、callable entry、CI 脚本、native baseline。

1. 提交的薄 Xcode runner，不用 CMake 生成 app 或 console runner

Runner 是 ios-runner/ 下的手写 Xcode 项目，含 app target 与 XCTest target。XCTest 提供逐 test 的 pass/fail、Xcode Test navigator 集成，以及「选 iPhone → Cmd-U」的一键真机流程。

  • 备选 — CMake 生成 iOS app + xctest_add_bundle：build 定义统一，但 CMake 的 iOS app / XCTest / device signing 支持很繁琐，与真机 workflow 冲突。否决。
  • 备选 — 通过 simctl spawn 跑 console binary：Simulator 上最简单，但没有干净的真机路径，也没有逐 test 粒度。否决。

2. 库通过 CMake 生成的 Xcode subproject 消费（fallback：直接编译源文件）

主路径：cmake -G Xcode 为 gaby_vm target 生成 Xcode 项目；runner 将其作为 subproject 引用并依赖其 library product。gaby_vm source list 仍以 CMake 为单一来源，Xcode 按所选 destination（sim arm64 / device arm64）构建依赖。

Fallback（若 subproject 在 sim/device 分裂上太 awkward）：runner 直接编译 gaby_vm 源文件，由 drift check 在 runner 的 source list 与 CMake 的 GABY_VM_IMPORTED_SOURCES / GABY_VM_SOURCES 不一致时 fail。

无论哪条路径，都不集成预编译 .a——那会迫使 per-SDK/arch lib 切换并增加 pre-Xcode build 步骤。由 Spike A 决定。

补充库 source list 的单一来源在 Sources/gaby_vm/src/CMakeLists.txt：GABY_VM_IMPORTED_SOURCES（VIXL 导入源）与 GABY_VM_SOURCES（项目自写源）分别 target_sources 进 gaby_vm。drift check 应对齐这两组变量。Sources/gaby_vm/src/CMakeLists.txt:12-58

3. Callable entry — 一套逻辑，两个 driver

每个 correctness suite 与 bench kernel 暴露可调用入口（如 int run_vixl_port_integer()、Report run_business(Mode)）。现有 host 可执行文件收成薄 main()；iOS XCTest case 与 bench runner 调用同一入口。iOS 不重复 test/bench 逻辑。

4. CI：correctness 门禁，perf 报告，skip 必须显式

  • ci/ios-test.sh 与 ci/ios-bench.sh 承载全部逻辑；workflow 只调用它们（与 docs/refs/ci.md 一致）。
  • Simulator 上的 correctness 是确定性的 → 作为 ci.yml 新 job ios-sim-test 门禁 PR。iOS bench 仅 report → bench.yml 里的一个 step。
  • 两个脚本都先检查 runner 架构。非 arm64（如无 arm64 simulator 的 x86 runner）→ 在 job summary 与 sticky comment 里打印醒目的 SKIPPED，exit 0。skip 必须被看见，不能静默 pass。

5. iOS 上的 native baseline：尽力而为，失败要大声说

Runner 是 development-signed 且 debugger-attached，因此 MAP_JIT 在 Simulator（host process）与 dev device 上都应可用——复用现有 bench/native_jit.entitlements plumbing。bench 在 baseline 可用时报告它；executable memory 被拒时明确报告「native baseline not run」，退化为 cache/decoder 对比。由 Spike B 验证。

## Decisions

### 1. A committed thin Xcode runner, not a CMake-generated app or a console runner

The runner is a hand-written Xcode project under `ios-runner/` with an app
target plus an XCTest target. XCTest gives per-test pass/fail, Xcode Test
navigator integration, and a one-click "select iPhone → Cmd-U" device flow.

- *Alternative — CMake-generated iOS app + `xctest_add_bundle`:* keeps one build
  definition but CMake's iOS-app / XCTest / device-signing support is fiddly and
  fights the device workflow. Rejected.
- *Alternative — console binary via `simctl spawn`:* simplest for the simulator
  but has no clean device story and no per-test granularity. Rejected.

### 2. Library consumed via a CMake-generated Xcode subproject (fallback: direct source compile)

Primary: `cmake -G Xcode` generates an Xcode project for the `gaby_vm` target;
the runner references it as a subproject and depends on its library product.
The gaby_vm source list stays single-sourced in CMake, and Xcode builds the
dependency for whichever destination (sim arm64 / device arm64) is selected.

Fallback, if the subproject proves awkward across the sim/device split: the
runner compiles the gaby_vm sources directly, guarded by a **drift check** that
fails when the runner's source list diverges from CMake's
`GABY_VM_IMPORTED_SOURCES` / `GABY_VM_SOURCES`.

Either way we do **not** integrate a prebuilt `.a` — that forces per-SDK/arch
lib juggling and a pre-Xcode build step. Decision gated by Spike A.

### 3. Callable entry points — one logic, two drivers

Each correctness suite and each bench kernel exposes a callable entry (e.g.
`int run_vixl_port_integer()`, `Report run_business(Mode)`). The existing host
executables collapse to thin `main()`s over those entries; the iOS XCTest cases
and bench runner call the same entries. No test/bench logic is duplicated for
iOS.

### 4. CI: correctness gates, perf reports, skip is explicit

- `ci/ios-test.sh` and `ci/ios-bench.sh` hold all logic; workflows only call
  them (matches [`docs/refs/ci.md`](../../../docs/refs/ci.md)).
- Correctness on the Simulator is deterministic → it **gates** the PR, as a new
  `ios-sim-test` job in `ci.yml`. iOS bench is report-only → a step in
  `bench.yml`.
- Both scripts check the runner arch first. Not arm64 (e.g. an x86 runner with
  no arm64 simulator) → print a loud `SKIPPED` line to the job summary and
  sticky comment, exit 0. The skip is surfaced, never a silent pass.

### 5. Native baseline on iOS: try it, degrade loudly

The runner is development-signed and debugger-attached, so `MAP_JIT` is expected
to work on both the Simulator (a host process) and a dev device — reusing the
existing `bench/native_jit.entitlements` plumbing. The bench reports the native
baseline when it is available and explicitly reports "native baseline not run"
when executable memory is denied, falling back to cache/decoder comparison.
Gated by Spike B.

Risks

subproject awkwardness、MAP_JIT、信号 handler 冲突、x86 skip 难测、iOS 13 + C++20。

  • CMake 生成的 Xcode subproject 对 sim+device 很 awkward → Spike A 先行；fallback 为 direct-source-compile + drift guard（Decision 2）。两条路都能 ship。
  • 某些 iOS 配置下 MAP_JIT 被拒 → native baseline 变为可选，缺失必须报告，不能静默丢弃（Decision 5）。
  • Crash-guard 信号 vs XCTest 自带 handler。 vixl_port harness 安装 SIGSEGV/SIGBUS/SIGILL/SIGABRT/SIGALRM handler；XCTest 也装自己的。→ Spike A 验证共存；若冲突，iOS driver 以 non-signal reporting 模式跑 suite。
  • x86 skip 路径在 arm64 runner 上无法 exercise。 → 架构检查保持纯逻辑、可在本地 dry-run，以便无 x86 机器也能测。
  • iOS 13 deployment floor + C++20。 库用 std::span / std::variant（header-only）。→ Spike A 确认 runner 在现有 iOS 13 floor 上能 build；只有被迫时才提高 floor，并记录。

## Risks / Trade-offs

- **CMake-generated Xcode subproject is awkward for sim+device** → Spike A first;
  fallback to direct-source-compile + drift guard (Decision 2). Either path
  ships.
- **`MAP_JIT` denied in some iOS config** → native baseline becomes optional and
  its absence is reported, not silently dropped (Decision 5).
- **Crash-guard signals vs XCTest's own handlers.** The vixl_port harness
  installs SIGSEGV/SIGBUS/SIGILL/SIGABRT/SIGALRM handlers; XCTest installs its
  own. → Verify coexistence in Spike A; if they conflict, the iOS driver runs
  the suite in a non-signal reporting mode.
- **The x86 skip path can't be exercised on an arm64 runner.** → Keep the arch
  decision a pure, locally dry-runnable check so it can be tested without an x86
  machine.
- **iOS 13 deployment floor + C++20.** The library uses `std::span` / `std::
  variant` (header-only). → Confirm the runner builds at the existing iOS 13
  floor during Spike A; raise the floor only if forced, and note it if so.

Migration

纯增量；建议顺序：抽 entry → 本地 Simulator → CI test → CI bench。

纯增量，不删除任何东西。建议顺序：

  1. 抽取 callable entry（host CTest 全程保持 green）
  2. 构建 runner，在本地 Simulator 上验证
  3. 把 ci/ios-test.sh 接入 ci.yml
  4. 把 ci/ios-bench.sh 接入 bench.yml

ios-sim-test job 可以先在分支上以 non-required 落地，在 main 上观察稳定后再 flip 为 required gate。

## Migration Plan

Additive; nothing is removed. Suggested order: extract callable entries (host
CTest stays green throughout) → build the runner and prove it on the Simulator
locally → wire `ci/ios-test.sh` into `ci.yml` → wire `ci/ios-bench.sh` into
`bench.yml`. The `ios-sim-test` job may land non-required on a branch first and
be flipped to a required gate once it is observed stable on `main`.
