1. Spikes — 定路径再固定结构

Spike A 验证 Xcode 集成与 iOS 兼容性；Spike B 验证 native baseline 在 iOS 上能否跑。

在提交 ios-runner/ 目录结构和 CI job 之前，先用 throwaway app 跑通关键假设。Spike A 覆盖库集成方式、crash-guard 与 XCTest 的 signal handler 共存、C++20 部署版本；Spike B 单独探 MAP_JIT / native baseline 在 Simulator 与真机上的可用范围。

  • 1.1Spike A：用 cmake -G Xcode 生成 gaby_vm Xcode 工程；在临时 app + XCTest 里引用，确认 iphonesimulator（arm64）和 iphoneos（arm64）都能 build 并 link。记录结果；若太别扭，改用 direct-source-compile fallback（design Decision 2）。
  • 1.2Spike A：确认 vixl_port 的 crash-guard signal handler（SIGSEGV/SIGBUS/SIGILL/SIGABRT/SIGALRM）在 Simulator 上与 XCTest 的 handler 能共存；若冲突，定义 iOS 非 signal 上报模式。验证故意崩溃的 test body 仍能被报告，不会静默丢失。
  • 1.3Spike A：确认 runner 在当前 iOS 13 部署版本下能用 C++20（std::span / std::variant）编译；记录实际使用的 floor，只有被迫时才提高。
  • 1.4Spike B：确认 MAP_JIT / native baseline 在 iOS Simulator 和 dev-signed、debugger-attached 真机上能否运行；记录哪里可用、哪里必须降级（design Decision 5）。

2. Callable entry points — host 全程保持绿灯

先写 failing test 驱动重构，再抽入口；每步都验证 host CTest 与 bench 不退化。

重构顺序是 TDD 式的：先加一个会失败的 host test 指向新 callable entry，再把 suite/bench 体抽出来，最后把现有可执行文件收成薄 main()。iOS runner 和 host driver 共用同一套实现，所以 host 侧必须始终 green。

  • 2.1在重构前，加一个会失败的 host test，通过新的 callable entry（例如 run_vixl_port_integer()）驱动一个 correctness suite。
  • 2.2把每个 correctness suite 的 body 抽成 callable entry；对应 CTest 可执行文件收成薄 main() 包装。验证 ctest --test-dir build/debug -R vixl_port 和完整 CTest 仍全绿。
  • 2.3把每个 bench kernel 抽成 callable entry；bench 可执行文件改成分发到该入口。验证 bench_business --verify 通过，bench 报告数值不变。
注意这是 iOS runner 的前置条件：XCTest 只能调用 entry，不能复制一份 suite 逻辑。

3. iOS runner project

提交 ios-runner/ Xcode 项目，XCTest 跑 suite 与 bench；Simulator 免签名，真机签名不进仓库。

按 Spike A 结论消费库（CMake 生成的 subproject，或 direct-compile + drift guard）。XCTest 逐项调用阶段 2 抽出的 entry；bench 路径输出 harness 报告，native baseline 按 Spike B 结果尝试或显式报告未运行。

  • 3.1创建 ios-runner/ Xcode 项目：Host App target + XCTest target，按 Spike A 结论消费库（generated subproject 或 direct-compile）。
  • 3.2若走 fallback 路径，加 source-list drift guard；验证当 CMake lists 里有文件但 runner 里缺失时会失败。
  • 3.3加 XCTest case，调用 correctness-suite entry（一个 suite → pass/fail）。验证 arm64 Simulator 上 xcodebuild test 跑全 suite 且全绿。
  • 3.4在 runner 里加 benchmark 路径，调用 bench entry 并输出 harness 报告；按 Spike B 尝试 native baseline，不行则显式 fallback 为「未运行」。验证能看到数值或明确的缺失说明。
  • 3.5签名：Simulator build 不需要 identity；真机签名来自 untracked xcconfig / automatic signing——不把 team 或 identity 提交进仓库。验证干净 checkout 在无签名配置下能 build Simulator。
