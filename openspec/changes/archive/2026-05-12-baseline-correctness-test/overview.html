baseline-correctness-test

给刚导入的 VIXL AArch64 simulator 织一张回归网 —— 用手工编码的 uint32_t 序列穿过 Simulator::RunFrom, 在 ADD/SUB/MUL、AND/ORR/EOR、LDR/STR、B.cond/CBZ/CBNZ、BL+RET 五个指令族上断言寄存器与内存状态。 smoke 只证明能跑;这个测试证明跑得对。

为什么需要这个测试

现有的 simulator_smoke 只验证一件事:能不能构造一个 Simulator 并跑一条 NOP 不崩。 它不检查 PC 是否前进、不检查寄存器状态、不检查任何语义。 对一份刚刚原样导入、还没改过一行的 VIXL 代码,这是"它在那里"的证明,不是"它对"的证明。

没有 simulator_correctness 时

• VIXL 上游 re-sync 后,leaf 语义可能静默改变,要等下游用户报 bug 才知道

• 未来的 predecode/dispatch cache 上线时,没有任何 baseline 可对照"cache 是不是改变了语义"

• 改 simulator 内部代码(refactor、移文件、改命名)只能靠"编译过就算赢"

• 新平台(iOS/arm64-linux)首次跑 simulator,无从分辨是平台问题还是代码问题

有 simulator_correctness 后

• 每次 re-sync、refactor、cache 改动都先要过这关,语义回归在 PR 阶段就被拦下

• cache 上线后,同一序列同时穿过两条路径,直接 diff 输出 = differential oracle

• 新平台只要 ctest 全绿就建立了第一道平台可信度

• 用户报 bug → 加一个 sub-test → 永久回归保护,模式化

具体覆盖什么

五个指令族,每个族至少一个子测试。每个子测试是一段独立的 uint32_t[] 序列,以 RET 收尾, 以"宿主算出的期望值"为对照标准。

Arithmetic
ADD · SUB · MUL
最基础的整数 ALU 通路。任何 leaf 通路退化都会在这里先被发现。
Logical
AND · ORR · EOR
位运算 leaf。操作数刻意选择互不相等的输出,防止编码互换的"巧合通过"。
Load / Store
LDR · STR
(imm + reg offset)
内存访问路径。买宿主侧 stack buffer,地址通过寄存器传入,两种寻址模式都覆盖。
Conditional
B.cond · CBZ · CBNZ
分支译码 + NZCV 通路。B.cond 前必置 SUBS,不依赖 reset 默认值。同时直接断言 ReadNzcv()。
Call / Return
BL · RET
函数调用 ABI 的最小单位。callee 写哨兵值 0xc011ab1e,外层 LR 用 X19 保住以正常终止。

测试家族中的位置

本次新增的 simulator_correctness 是项目第三个 CTest case。 它在 simulator_smoke 之上多了一层"语义层"覆盖。

三个测试目标,层层叠加。本次新增填上"语义正确性"这一层。

每个序列的形状(D3 + D7)

关键不变量:Simulator::ResetRegisters() 把 LR 初始化为 kEndOfSimAddress (= nullptr), Run() 在 pc == kEndOfSimAddress 时退出。 所以每段序列都是一个隐式的"叶子函数体",以 RET 收尾。

每个 sub-test 的生命周期:写入 → 跑 → 终止于 RET 到 NULL LR → 断言。

11 条设计决策

点击展开任何一条查看完整理由与备选方案。完整内容见 design.md。

D1单 TU + 按指令族分子测试 + 手写断言

一份 test/simulator_correctness.cc;一个 TestState 结构 + 三个 expect_eq_* 助手;五个 family 函数;一个 main()。

为什么: 匹配现有 simulator_smoke 的纯手工模式;不引入 Catch2/GoogleTest 这类依赖,因为测试面积太小,框架成本不值得;真到几十个子测试再说。

D2指令序列直写 alignas(uint32_t) uint32_t[],每 word 配助记符注释

每个编码值在源码中以字面量出现,旁边一行注释写明助记符。作者侧用 llvm-mc 校验后手抄进来。

为什么: Tier-0 边界禁止 assembler 进仓库;助记符宏 = 30 行的 assembler,等同绕过。外部 llvm-mc 只在写代码时使用,产出落地为字面量,运行链路里没有 assembler。

D3终止协议:序列末尾必为 RET,依赖 LR == NULL 初始不变量

ResetRegisters() 将 LR 初始化为 kEndOfSimAddress (nullptr);Run() 在 pc == kEndOfSimAddress 时退出。所有序列都是隐式叶子函数。

为什么: 这就是 VIXL 自带的协议;镜像它而不是发明 HLT/BRK trap,既不动 imported 代码也无需注册 callback;末尾 RET 还顺带验证 RET leaf 的语义。

D4用裸 WriteXRegister / RunFrom / ReadXRegister,不用模板版 RunFrom<R, Args...>

显式写入输入寄存器 → 调用非模板 RunFrom → 显式读取输出寄存器。

为什么: 模板版门控在 VIXL_HAS_ABI_SUPPORT + C++11 + GCC 版本检查上,任何一个失败就整组测试消失;且模板只能传/读 ABI 寄存器(stack args 是 TODO),无法检查任意寄存器/NZCV。

D5每个子测试构造一份新的 Simulator

不在 family 内复用 Simulator + ResetState(),而是局部构造、自然析构。

为什么: 避免"忘了重置某个状态"的二阶 bug;Simulator 内部状态相当庞大,reset 覆盖度漂移会被这个选择隔绝;构造开销在 CI 里看不见。

D6Load/Store 用宿主侧 stack buffer,地址通过 X 寄存器传入

子测试内 alignas(uint64_t) std::array<uint8_t, 64> buf;WriteXRegister(0, &buf[0]);序列内通过 [X0, #imm] 或 [X0, X1] 访问。

为什么: Simulator 是软件解释器,没有 guest 虚拟地址空间;"内存"就是宿主进程内存;stack 分配天然生命周期对齐 sub-test。

D7BL+RET 用 MOV X19, LR / BL / MOV LR, X19 / RET 保住外层 LR

callee 被 BL 调用时会覆盖 LR;若不恢复,外层 RET 就跳不回 NULL,Run() 不会终止。用 X19(callee-saved)做中转。

为什么: 比 STP X29,X30,[SP,#-16]! / LDP 的标准 prologue 简单(不需要 SP 算术和成对 load/store,那是另一族指令);保持本次测试自包含。

D8B.cond 必须前置 SUBS/CMP 显式设置 NZCV

不依赖 ResetSystemRegisters 给的 NZCV 默认值;B.cond 子测试同时直接断言 ReadNzcv() 的 Z 位。

为什么: 把"flag-setting 通路对不对"和"条件解码对不对"分开:只看 marker 寄存器的话,这两类回归会混在一起;同时断言 NZCV 直接区分。

D9可执行文件和 CTest case 都叫 simulator_correctness(不带 _test 后缀)

对齐 simulator_smoke 命名:两者(可执行 + CTest case)同名。

为什么: Proposal 里写 "e.g. simulator_correctness_test" 是举例;设计阶段择优定名,与现有项目惯例一致。

D10CMake 块逐行抄 simulator_smoke 的块

PRIVATE 链接 gaby_vm::gaby_vm;PRIVATE include ${PROJECT_SOURCE_DIR}/src;三个 VIXL 编译定义;gaby_vm_apply_compile_flags(项目策略助手,不是 imported 放宽助手)。

为什么: 测试代码对 imported 头的访问权限和 smoke 完全一致;未来 re-sync 改 VIXL 定义时,两个块一起改、grep 友好;测试代码受项目正常 warning 策略约束。

D11输出流默认 stdout,不做 trace 调整

同 simulator_smoke;不调 set_trace_parameters。CTest 捕获 stdout 仅在失败时打印,所以默认 verbosity 在通过时无害,在失败时反而有用。

这套 harness 在后续开发里会怎样持续发挥作用 ★

这是写这个 PR 之外、文档里最容易被忽略的事 —— 这一份 ~300 行 的测试代码不是为了"现在过 CI",而是为了在接下来的几年里反复 catch 各种回归。 下面是六个具体的未来场景。

项目主目标

1. Predecode / dispatch cache 的 differential oracle

触发事件:项目核心优化方向落地 —— 把 decode → visit → leaf 这条解释路径替换/旁路为 cached → direct leaf dispatch(CLAUDE.md 里写的正是这条线)。

本次新增的 simulator_correctness 此时成为差分参照 —— cache 路径不能比 visitor 路径"多做一件事或少做一件事"。

没有它会怎样:cache 上线只能靠"smoke 还跑得过 + 单元 microbench 看起来变快"来判断没坏 —— 这两个证据都不证明 leaf 语义保持。一个改写了 ADD 上溢行为的 cache bug 可能要等几个月才被下游程序触发。

有它会怎样:cache 实现的 PR 在测试阶段就跑 simulator_correctness 两遍(forced visitor / forced cached), 任何状态 diff 立即失败,问题精确定位到指令族。 随着 cache 覆盖更多 leaf,这个测试集合同步扩展,差分覆盖与 cache 覆盖共生长。

2. VIXL 上游 re-sync 时的语义漂移检测

触发事件:团队从 ../vixl 拉一次新提交,合并到 src/ 下的 imported 文件。

没有它会怎样:上游修了一个 MUL overflow bug,改变了我们以前依赖的行为; 没有任何 CI 信号能区分"合并干净"和"语义改了"。

有它会怎样:re-sync PR 里 simulator_correctness 的 MUL 子测试失败 → bug 在 review 阶段被讨论,而不是几周后由下游用户报回来。同样适用于 NZCV、移位、load/store 边界等所有上游可能变动的 leaf。

3. 覆盖扩展:SIMD / FP / SVE / PointerAuth

触发事件:未来某个 change 决定加 FP 算术覆盖。

没有它会怎样:那个 change 从零搭一套测试框架、断言助手、Simulator 工厂、CMake 装配。

有它会怎样:直接在文件里加 run_fp_tests(TestState&), 复用 make_simulator / expect_eq_* / 编码校验工作流 / CMake 块。 哈尼斯本身就是反复增量复用的资产 —— 固定成本只付一次。

4. Bug 复现成回归保护

触发事件:有人报告 "Simulator 跑到 SUB x0,x1,x2 当 x1=0x8000...000 时算错"。

没有它会怎样:修 bug + 写一个一次性脚本验证;一段时间后没人再跑这个脚本,同一 bug 又回来。

有它会怎样:修 bug 时必须在 run_arithmetic_tests 加一行 sub-test 复现该输入; bug 自此永久回归保护。模式化:"接到 bug → 加 sub-test → 修 → 永远绿"。

5. iOS / Android / arm64-linux 移植首信号

触发事件:项目第一次在 iOS 设备上 build + run 这套代码。

没有它会怎样:build 过了就以为没事;实际 LDR 在某种对齐下因平台差异行为不同; 要靠下游用户在生产里命中才发现。

有它会怎样:第一次在新平台跑 ctest,如果 simulator_correctness 全绿, 就建立了"在这个平台上整数核心通路也对"的可信度;失败时也能立刻定位到指令族而不是茫茫代码海。

6. 工具链 / 编译器升级的保险

触发事件:clang 17 → 19、libc++ 升级、macOS SDK 切版。

没有它会怎样:新编译器对某段 UB 行为优化得不同,Simulator 中某个 leaf 路径被影响; 你以为只是升级,实际改了 simulator 行为,要靠下游回归发现。

有它会怎样:升级 PR 上 simulator_correctness 任何子测试一变红, 升级讨论从"要不要升"自然转入"哪条路径变了"。

这套 harness 的"复利"模型

写一次,赚一辈子。每加一个新指令族(SIMD/FP/SVE 等),固定成本(harness 装配、Simulator 工厂、断言助手、编码校验流程)都不需要再付一次,只付增量(那一族的子测试)。 每发生一次未来事件(re-sync / cache 上线 / 平台移植 / 工具链升级 / bug 报告),simulator_correctness 都是第一个证据源。

这就是为什么这个 300 行 的测试值得 4 份 artifact 的规划工作 —— 它的杠杆来自后续每一次复用与每一次拦下回归,而不是它本身那一次 CI 通过。
