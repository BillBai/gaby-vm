为什么要做这个 change

项目的优化方向 predecode once → cached dispatch → repeated execution 只有在有可测量基准的前提下才有意义。方法论文档 docs/refs/baseline-benchmark-suite.md 已经写好但没有可跑的代码，前一个 archived 改动 baseline-correctness-test 也把吞吐量测量明确推迟到了这次。

这次 change 把 harness 落地——workload、计时循环、报告格式——让所有后续优化（cache、leaf 微调、任何事）都有一个 measurement contract 可以对齐。

7 个 Design Decision

Decision 1

Workload 是 commit 的 C++ header

每个 workload 一个头文件，包含 4 个 constexpr： kXxxWorkloadInstructions[]、 StaticWordCount、 DynamicInstructionsPerIteration、 GeneratorTag。schema 在两份 workload 间统一。

为什么不用二进制 blob：免 I/O、免路径解析、对 iOS 嵌入零摩擦。
Decision 2

Workload 离线生成、不进 build

Mixed：仓库外的 C++ 程序用上游 BenchCodeGenerator 生成字节，同时跑一次 instrumented simulator（set_trace_parameters hook）记录动态指令数。
Smoke：llvm-mc + llvm-objcopy，无上游 VIXL 依赖。

build 保持 hermetic；重新生成是开发者偶发动作，可重现（seed + commit hash）。
Decision 3

时间预算循环 + 数 iteration

1 次 warm-up（丢弃）→ steady_clock 起点 → 循环 RunFrom 直到累计时长 ≥ 目标。默认 5 秒（匹配上游 kDefaultRunTime），可用 --seconds <float> 覆盖。

和上游 bench-mixed-sim 形状逐字对齐，跨对比直接可信。
Decision 4

计时用 std::chrono::steady_clock

单调、亚微秒分辨率、零平台分支。多秒测量窗口下分辨率冗余，rdtsc 这种带不来增益。

不复制上游 BenchTimer（它在 Tier 0），<chrono> 够用。
Decision 5

V1：一个 workload 一个 binary

两个 binary（bench_baseline、bench_smoke）共享一份 bench_runner（OBJECT library）；只有 main() 里 include 的 workload header 不同。

N=2 还不值得搞 --workload <name> 注册表，复制 20 行 main 比设计选择 UX 便宜。
Decision 6

输出 = 纯文本 K/V 行

主指标 iterations_per_second（=upstream PrintResults 格式）；派生 throughput_insn_per_sec 和 ns_per_instruction 都用动态计数除，绝不用静态字数。

JSON 延后到有下游消费者再上；K/V 文本 grep/awk 友好。
Decision 7

CMake mirror test/CMakeLists.txt

OBJECT 库 bench_runner + 两个 add_executable。不注册 CTest：benchmark 是开发者主动跑的，不归 ctest 管。

复用已经验证好的 include + define pattern；不污染正确性测试套。

Risks & Trade-offs（6 条）

!
Workload-blob 与导入的 simulator 漂移 — workload 是按某个 upstream commit 生成的；如果之后导入的 Tier 1-3 又落后了，可能出现 simulator 不认的指令编码。 缓解：header 的 kXxxWorkloadGeneratorTag 记录上游 commit hash；README 说明对齐期望。
!
不严控宿主就有 wall-clock 噪声 — Laptop turbo / busy desktop / CI runner 会让数字晃。 缓解：接受。V1 目标是数量级正确，不是发表级精度。README 只写一句"别用电池、找个空闲机器"。
!
RunFrom 调用站开销在小 workload 上会显著 — 当前 workload 都够大（≥ 32 条且循环数百万次），可以忽略。未来加小 workload 时需重新评估。 缓解：单条 workload 大小是 header 里的事，加新 workload 的 reviewer 自觉考虑。
!
Header-blob 把 workload 大小锁成编译期常量 — 改大小需要重新生成 + 重新编译。V1 不是问题（workload 形状固定）。 缓解：等第二个 workload 进来时再评估是否切换到运行期文件加载。
!
默认 --seconds 5 在某些主机上不合适 — 50 ms/iter 的慢机器 5 秒只跑 ~100 次（噪声高）；0.5 ms/iter 的快机器是 ~10000 次（够了，略浪费）。 缓解：是 CLI flag，不是硬编码常量。匹配上游 kDefaultRunTime。
!
记录的动态计数与实际执行漂移 — 动态计数是 commit 进 header 的；如果导入的 simulator 行为偏离上游，记录值会失准，ns/insn 会错（但 iterations/sec 仍对）。 缓解：主指标用直接观测的 iterations/sec；ns/insn 在输出里标 "derived" 提醒读者。
!
两个 workload 角色不同容易被误用 — 把 bench_smoke 的数字当 baseline 发出去，那就完了。 缓解：spec 用 workload-无关的语言写契约；README 明确说"哪个 binary 负责报数"。

Open Questions（3 个，全部 resolved resolved）

Q1 — Mixed workload 离线计数用什么 hook？
选 B：用 simulator 自带的 set_trace_parameters hook（配置为"计数不打印"）；不写自定义 DecoderVisitor。理由：hook 已经接好，几行配置代码即可，比写一个新 visitor 类更省事。
Q2 — 256 KiB / 64 Ki 这个 buffer 尺寸合适吗？
保留：V1 跟上游 bench-mixed-sim 用一样的 256 KiB。iOS 适配尺寸延后到真的有 iOS 嵌入目标的 change 再决定。
Q3 — 要不要 ship 一个 smoke workload？
做：提升到 scope 里，成为第二个 binary bench_smoke。直线流水 ~32 条指令，llvm-mc 离线生成，无上游 VIXL 依赖。能在 mixed workload 还没生成前先验证 harness 接线正确。
