benchmark-harness · 规格

让 benchmark harness 可被 iOS runner 等非 CTest driver 调用，并在 native baseline 不可用时显式说明而非静默省略。

specs/benchmark-harness/spec.md OpenSpec delta spec EN → 中文

本文要点

  1. 每个 benchmark workload/kernel SHALL 通过 callable entry 暴露，供 iOS runner 等非 bench 可执行文件的 driver 调用。
  2. 现有 bench 可执行文件 SHALL 转发到同一 entry point，harness 逻辑不得按 driver 重复实现。
  3. 当 native baseline 因 executable memory 不可用而无法测量时，harness SHALL 显式报告「native 未运行」，并继续输出 cache/decoder 结果。
  4. 当 executable memory 可用时，native baseline 的测量与报告行为与现有 harness 一致。

Callable entry 驱动 kernel

Bench 可执行文件与 iOS runner 共用同一套 kernel entry。

Benchmark harness SHALL 为每个 workload/kernel 暴露 callable entry point，供 bench 可执行文件 main() 以外的 driver（例如 iOS runner）使用。现有 bench 可执行文件 SHALL 转发到这些 entry point，harness 逻辑 SHALL NOT 按 driver 重复实现。

A non-CTest driver runs a kernel
WHEN bench 可执行文件以外的 driver 调用某个 kernel 的 entry point
THEN 它运行相同 workload 并产出与对应 bench 可执行文件相同的 metrics
Bench executables dispatch to the shared entry
WHEN bench 可执行文件运行某个 workload
THEN 它转发到 shared entry point，而非重新实现 workload body

### Requirement: Benchmark kernels are drivable through callable entry points

The benchmark harness SHALL expose each workload/kernel through a callable
entry point usable by a driver other than a bench executable's `main()` (for
example the iOS runner). The existing bench executables SHALL dispatch to the
same entry points, so the harness logic is not duplicated per driver.

#### Scenario: A non-CTest driver runs a kernel

- **WHEN** a driver other than a bench executable calls a kernel's entry point
- **THEN** it runs the same workload and produces the same metrics as the corresponding bench executable

#### Scenario: Bench executables dispatch to the shared entry

- **WHEN** a bench executable runs a workload
- **THEN** it dispatches to the shared entry point rather than reimplementing the workload body

Native baseline 缺失须显式报告

无法获取 executable memory 时明确说明 native 未跑，cache/decoder 照常输出。

Harness SHALL 在 native baseline 缺失时显式报告，而非静默省略：当被要求测量 native baseline 但 native 执行所需的 executable memory 不可用时，它 SHALL 报告 native baseline 未运行，并继续报告可用的 cache、decoder 模式结果。

Native baseline unavailable
WHEN 无法获取 native 执行所需的 executable memory
THEN harness 输出中包含 native baseline 未运行的明确说明
AND cache 与 decoder 的测量结果仍然产出
Native baseline available
WHEN native 执行所需的 executable memory 可用
THEN native baseline 被测量并报告，行为与此前一致

### Requirement: A missing native baseline is reported explicitly

The harness SHALL report a missing native baseline explicitly rather than
silently omitting it: when it is asked to measure the native baseline but
executable memory for native execution is unavailable, it reports that the
native baseline was not run and continues reporting the available modes
(cache, decoder).

#### Scenario: Native baseline unavailable

- **WHEN** native execution memory cannot be obtained
- **THEN** the harness output contains an explicit indication that the native baseline did not run
- **AND** the cache and decoder measurements are still produced

#### Scenario: Native baseline available

- **WHEN** native execution memory is available
- **THEN** the native baseline is measured and reported as before
