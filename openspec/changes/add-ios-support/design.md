## Context

See [`proposal.md`](proposal.md) for motivation. Today the only execution
target exercised by CI and the test/bench suites is macOS arm64. The library
already cross-compiles for iOS arm64; what is missing is a way to *run* the
suites on iOS and a CI path that does so. CI structure and the deterministic-
gate philosophy are documented in
[`docs/refs/ci.md`](../../../docs/refs/ci.md); the test harness in
[`docs/testing.md`](../../../docs/testing.md).

Two facts shape every decision below:

- The existing tests and benches are standalone `main()` executables driven by
  CTest (host-only). iOS has no equivalent of "run this console binary".
- The iOS runner is always a **development** build, launched from Xcode with a
  debugger attached — the configuration in which iOS permits `MAP_JIT`.

## Goals / Non-Goals

**Goals:**

- Run the full correctness suites and the benchmark harness on iOS — the
  Simulator under CI, a physical device locally.
- Keep one source of truth for both the library source list and the suite/bench
  logic; the iOS path must not fork either.
- Fit the existing CI model: deterministic correctness gate, report-only perf,
  all logic in portable `ci/` scripts.

**Non-Goals:**

- Automated device runs in CI (local-only).
- Changing interpreter semantics, the public API, or the VIXL import boundary.
- A reusable iOS app framework for embedders — the runner is a test/bench host,
  not a shipping artifact. tater integrates the library as source (submodule).

## Decisions

### 1. A committed thin Xcode runner, not a CMake-generated app or a console runner

The runner is a hand-written Xcode project under `ios-runner/` with an app
target plus an XCTest target. XCTest gives per-test pass/fail, Xcode Test
navigator integration, and a one-click "select iPhone → Cmd-U" device flow.

- *Alternative — CMake-generated iOS app + `xctest_add_bundle`:* keeps one build
  definition but CMake's iOS-app / XCTest / device-signing support is fiddly and
  fights the device workflow. Rejected.
- *Alternative — console binary via `simctl spawn`:* simplest for the simulator
  but has no clean device story and no per-test granularity. Rejected.

### 2. CMake generates the library project; XcodeGen generates the host; a script ties them (resolved by Spike A)

Spike A confirmed the library builds clean for **both** SDKs via the CMake
Xcode generator: `iphonesimulator` arm64 (`platform 7`, minos 14.0 — the
inherent Apple-silicon-simulator floor) and `iphoneos` arm64 (`platform 2`,
minos 13.0), no signing needed for the static lib.

The library's Xcode project is therefore **generated by CMake** (`cmake -G
Xcode`), keeping the source list single-sourced in `CMakeLists.txt` with no
drift guard. The host (app + XCTest) is generated by **XcodeGen** from a
committed `project.yml`, and references the CMake-generated `gaby_vm.xcodeproj`
as an external project, depending on its `gaby_vm` target so Xcode builds the
library per destination (sim arm64 / device arm64).

A single committed script (`ios-runner/generate.sh`) runs `cmake -G Xcode`,
then strips the legacy `PBXBuildStyle` objects CMake emits (XcodeGen's project
parser rejects them; Xcode itself does not need them — a structural plist edit,
not a regex hack), then `xcodegen generate`. A fresh clone is one command away
from an openable `ios-runner/GabyRunner.xcodeproj`. Both generated `.xcodeproj`s
stay out of git (regenerated by the script); the committed sources of truth are
`CMakeLists.txt` (library) and `ios-runner/project.yml` (host).

We do **not** integrate a prebuilt `.a` — the host depends on the library
target and Xcode builds it per destination.

### 3. Callable entry points — one logic, two drivers

Each correctness suite and each bench kernel exposes a callable entry (e.g.
`int run_vixl_port_integer()`, `Report run_business(Mode)`). The existing host
executables collapse to thin `main()`s over those entries; the iOS XCTest cases
and bench runner call the same entries. No test/bench logic is duplicated for
iOS.

### 4. CI: correctness gates, perf reports, skip is explicit

- `ci/ios-test.sh` and `ci/ios-bench.sh` hold all logic; workflows only call
  them (matches [`docs/refs/ci.md`](../../../docs/refs/ci.md)).
- Correctness on the Simulator is deterministic → it **gates** the PR, as a new
  `ios-sim-test` job in `ci.yml`. iOS bench is report-only → a step in
  `bench.yml`.
- Both scripts check the runner arch first. Not arm64 (e.g. an x86 runner with
  no arm64 simulator) → print a loud `SKIPPED` line to the job summary and
  sticky comment, exit 0. The skip is surfaced, never a silent pass.

### 5. Native baseline on iOS: try it, degrade loudly

The runner is development-signed and debugger-attached, so `MAP_JIT` is expected
to work on both the Simulator (a host process) and a dev device — reusing the
existing `bench/native_jit.entitlements` plumbing. The bench reports the native
baseline when it is available and explicitly reports "native baseline not run"
when executable memory is denied, falling back to cache/decoder comparison.
Gated by Spike B.

## Risks / Trade-offs

- **CMake-generated Xcode subproject is awkward for sim+device** → Spike A first;
  fallback to direct-source-compile + drift guard (Decision 2). Either path
  ships.
- **`MAP_JIT` denied in some iOS config** → native baseline becomes optional and
  its absence is reported, not silently dropped (Decision 5).
- **Crash-guard signals vs XCTest's own handlers.** The vixl_port harness
  installs SIGSEGV/SIGBUS/SIGILL/SIGABRT/SIGALRM handlers; XCTest installs its
  own. → Verify coexistence in Spike A; if they conflict, the iOS driver runs
  the suite in a non-signal reporting mode.
- **The x86 skip path can't be exercised on an arm64 runner.** → Keep the arch
  decision a pure, locally dry-runnable check so it can be tested without an x86
  machine.
- **iOS 13 deployment floor + C++20.** The library uses `std::span` / `std::
  variant` (header-only). → Confirm the runner builds at the existing iOS 13
  floor during Spike A; raise the floor only if forced, and note it if so.

## Migration Plan

Additive; nothing is removed. Suggested order: extract callable entries (host
CTest stays green throughout) → build the runner and prove it on the Simulator
locally → wire `ci/ios-test.sh` into `ci.yml` → wire `ci/ios-bench.sh` into
`bench.yml`. The `ios-sim-test` job may land non-required on a branch first and
be flipped to a required gate once it is observed stable on `main`.

## Open Questions

- Simulator destination selection: a pinned device/OS name vs a generic
  `platform=iOS Simulator` destination. Resolve in implementation.
- Whether device benchmark numbers get a committed reference doc under
  `docs/refs/` like the macOS baselines, or stay ad-hoc until they stabilise.
